## Executive Summary

**Overall health: YELLOW** — no P0 static/build issues, but the admin-domain logic review surfaced 4 P0s around admin-lockout + payout dual-control, and npm audit found a high-severity axios SSRF cluster.

### Top 5 by severity

1. **P0 — Admin lockout possible via `update_user_status`** — [backend/app/services/admin_service.py:194-208](backend/app/services/admin_service.py#L194-L208). Last-admin guard only runs in `toggle_superuser`; a single admin can suspend/deactivate every other admin (and themselves) via `update_user_status` with zero checks, leaving the platform with zero active admins.
2. **P0 — Payout dual-control bypassable by splitting** — [backend/app/services/payout_approval_service.py:74-97](backend/app/services/payout_approval_service.py#L74-L97). Threshold check is per-escrow-row. Admin who controls multiple smaller funded escrows for the same freelancer can release an unlimited total without second approval.
3. **P0 — Last-admin check race condition** — [backend/app/services/admin_service.py:222-232](backend/app/services/admin_service.py#L222-L232). `toggle_superuser`'s admin-count SELECT has no `FOR UPDATE`; two admins concurrently demoting each other both pass the check, both commit, zero admins remain.
4. **P1 — Axios SSRF + DoS + cloud-metadata exfiltration** — `frontend/package.json` pins axios 1.12.2. 3 CVEs fix at 1.15.2 (non-major bump): GHSA-3p68-rc4w-qgx5 (SSRF via NO_PROXY bypass), GHSA-fvcv-3m26-pcqx (cloud metadata exfiltration via header injection), GHSA-43fc-jf86-j433 (DoS via __proto__). **Upgrade blocker is trivial.**
5. **P1 — Audit-log loss race on admin mutations** — [backend/app/api/v1/endpoints/admin.py:107-115](backend/app/api/v1/endpoints/admin.py#L107-L115). `update_user_status` commits the mutation, THEN writes the audit row and commits again. Audit-write failure = permanent status change with no trail. Same shape in `toggle_admin`, `toggle_support`, `unsuspend_chat`.

## Layer 9: Logic review — admin domain (Saturday)

### P0 — Critical (must fix before next release)

- **[backend/app/services/admin_service.py:194-208](backend/app/services/admin_service.py#L194-L208)** — `update_user_status` does not protect the last admin. A single admin can suspend/deactivate the only other admin, then themselves, leaving the platform with zero admins. No `acting_admin` parameter, no self-target guard. *Combined with the bypass below this lets a hostile or careless admin lock everyone out.*
- **[backend/app/services/admin_service.py:222-232](backend/app/services/admin_service.py#L222-L232)** — Last-admin check counts `is_superuser AND status=ACTIVE`, but `update_user_status` can flip an admin to `SUSPENDED/DEACTIVATED` with no such check. The "last admin" invariant must be enforced on **every path** that drops an admin out of the active pool, not just on `toggle_superuser`.
- **[backend/app/services/admin_service.py:222-232](backend/app/services/admin_service.py#L222-L232)** — Last-admin check is not atomic. Two admins concurrently demoting each other both pass `admin_count > 1`, both commit, zero admins remain. Needs `with_for_update()` on the admin rows or a unique-row lock pattern.
- **[backend/app/services/payout_approval_service.py:74-97](backend/app/services/payout_approval_service.py#L74-L97)** — Dual-control threshold uses `<=` on a single escrow's amount with no aggregation. An admin who controls multiple smaller funded escrows for the same freelancer can release an unlimited total below threshold without ever needing a second approver. Only per-row enforcement, not per-time-window or per-payee.

### P1 — High

- **[backend/app/api/v1/endpoints/admin.py:235-248](backend/app/api/v1/endpoints/admin.py#L235-L248)** — `PUT /admin/jobs/{job_id}/status` writes no audit log entry. Job moderation is silent on the audit trail. Compliance/forensics gap.
- **[backend/app/services/admin_service.py:194-208](backend/app/services/admin_service.py#L194-L208)** — `update_user_status` accepts free-form `new_status: str` and coerces via `UserStatus(new_status)` with no try/except. Invalid value surfaces as raw 500 `ValueError` instead of `BadRequestError`. Same pattern at `update_job_status` (line 336-350) and `list_users` (line 170-173).
- **[backend/app/services/payout_approval_service.py:153-200](backend/app/services/payout_approval_service.py#L153-L200)** — `approve()` checks `requested_by_id == admin.id` (four-eyes) but does NOT verify the approving admin still has `is_superuser=True` at decision time. Route dependency catches it today, but service has no defense-in-depth.
- **[backend/app/services/audit_service.py:50-77](backend/app/services/audit_service.py#L50-L77)** — Audit writes swallow exceptions inside the **same `db` session** the caller will commit. A flush failure leaves the session partially failed; the caller's subsequent commit may silently lose the underlying admin action. Audit-on-success-only becomes audit-then-corrupt-the-action.
- **[backend/app/api/v1/endpoints/admin.py:107-115](backend/app/api/v1/endpoints/admin.py#L107-L115)** — `update_user_status` commits the mutation, THEN writes audit + commits again. Audit failure = permanent status change with no log. Same shape in `toggle_admin`, `toggle_support`, `unsuspend_chat`.
- [backend/app/services/payout_approval_service.py:295](backend/app/services/payout_approval_service.py#L295) — `decided_by_email=None` hardcoded in `list_pending`. Correct for pending rows, but if this endpoint ever returns approved/rejected, the field silently always-null.
- [backend/app/api/v1/endpoints/admin.py:194-212](backend/app/api/v1/endpoints/admin.py#L194-L212) — `unsuspend_chat` audits with `USER_STATUS_CHANGED` instead of a dedicated action (intentional per the inline comment); easy to overlook in alerting that filters on the enum.

### P2 — Medium

- [backend/app/services/admin_service.py:354-420](backend/app/services/admin_service.py#L354-L420) — `list_funded_escrows` returns full freelancer email + phone + qi_card details for every funded escrow with **no pagination cap**. A leaked admin cookie can scrape every freelancer's payout PII in one request.
- [backend/app/services/payout_approval_service.py:71-72](backend/app/services/payout_approval_service.py#L71-L72) — `amount_iqd = float(escrow.amount)` and threshold compare via float. Millions-of-IQD amounts are fine today but introduces a silent precision boundary. Use Decimal consistently.

- [backend/app/api/v1/endpoints/admin.py:441-487](backend/app/api/v1/endpoints/admin.py#L441-L487) — `/admin/audit-logs` gated on `get_current_staff` (support included). - [backend/app/api/v1/endpoints/admin.py:563-607](backend/app/api/v1/endpoints/admin.py#L563-L607) — `list_support_conversations` documents scoping to "own + unassigned" but `claim/release/resolve/reopen` (lines 615-679) rely on `MessageService` to enforce ownership; the endpoints don't re-check.
