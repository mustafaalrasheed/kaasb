## Executive Summary

Overall health: **YELLOW** — no immediate data-loss crashes but three P1 bugs affect the dispute system's correctness (dual-system blind spots, dead code, missing notifications on resolve), plus a high-severity axios vulnerability in the frontend.

Top 5 findings:

1. **P1** — `catalog_service.py:819` — Dual dispute system: disputes raised via `POST /services/orders/{id}/dispute` (CatalogService path) never create a `Dispute` model record. Admin panel at `GET /disputes` is therefore **blind** to all disputes from the primary UI path; they only appear in the deprecated `/services/admin/disputes` list.
2. **P1** — `catalog_service.py:819` — Unreachable dead code: `if order.status == ServiceOrderStatus.DISPUTED: raise BadRequestError(...)` is logically unreachable because `DISPUTED` is not in `allowed_statuses`. The earlier check catches every non-allowed status first, so the dedup guard never fires.
3. **P1** — `dispute_service.py:181–218` — `DisputeService.resolve_dispute` delegates to `catalog_svc.resolve_dispute` (which notifies both parties), then commits the Dispute record state change — but if the inner `catalog_svc` call succeeds and commits while the outer commit fails (e.g. network error), the Dispute row is stuck in OPEN/UNDER_REVIEW while the order is already COMPLETED/CANCELLED. Silent inconsistency.
4. **P1** — `frontend/src/lib/api/services.ts:101` — `servicesApi.reject` sends `reason` as a **query parameter** (`params: { reason }`) but the backend's `reject_service` endpoint reads it via `Query(...)`. This is architecturally fine, but query parameters are logged in access logs — rejection reasons (which may mention freelancer misconduct details) are exposed in plain-text server logs.
5. **P1** — `npm audit` — axios 1.12.2 has 3 CVEs: DoS via `__proto__` key (`GHSA-43fc-jf86-j433`), SSRF via NO_PROXY normalization bypass (`GHSA-3p68-rc4w-qgx5`), and cloud metadata exfiltration via header injection (`GHSA-fvcv-3m26-pcqx`). Fix: `axios@1.15.2`.

## Layer 9: Logic Review — Disputes

### Critical Findings

**`backend/app/services/catalog_service.py:819` — P1 — Dead-code double-dispute guard**

```python
if order.status not in allowed_statuses:
    raise BadRequestError(...)  # catches everything not in allowed set
if order.status == ServiceOrderStatus.DISPUTED:  # UNREACHABLE — DISPUTED not in allowed_statuses
    raise BadRequestError("A dispute has already been raised for this order")
```

`DISPUTED` is not in `allowed_statuses` (lines 808–812), so the first check always fires before the second. The duplicate-dispute guard never executes. A concurrent second call between the status-check and the `order.status = DISPUTED` assignment could theoretically create two disputes via a race condition (no `SELECT FOR UPDATE`).

**`backend/app/services/catalog_service.py:792` and `backend/app/services/dispute_service.py:30` — P1 — Dual dispute systems are mutually blind**

Two fully separate dispute flows exist:

- `POST /services/orders/{id}/dispute` → `CatalogService.raise_dispute` → sets `ServiceOrder.status=DISPUTED` + `Escrow.status=DISPUTED` but creates **no `Dispute` model record**.
- `POST /disputes/orders/{id}` → `DisputeService.open_dispute` → creates a `Dispute` record + sets order/escrow status.

The admin `/disputes` panel (`GET /disputes`) only queries the `disputes` table (the Dispute model). All disputes raised via the primary UI path (the catalog endpoint) are **invisible** to this panel. Admin resolution via `POST /disputes/{id}/resolve` can only resolve disputes that went through the F5 path. Disputes from the legacy path can only be resolved via `POST /services/admin/orders/{id}/resolve-dispute`.

**`backend/app/services/dispute_service.py:197–216` — P1 — Silent state divergence on resolve commit failure**

`DisputeService.resolve_dispute` calls `catalog_svc.resolve_dispute(...)` which internally commits (order + escrow updated, notifications queued), then returns. If the subsequent `await self.db.commit()` (which updates `Dispute.status`) fails, the order is COMPLETED/CANCELLED and the escrow is settled, but the `Dispute` record remains in `OPEN` or `UNDER_REVIEW`. The admin panel shows the dispute as unresolved while the order is done. No rollback of the already-committed catalog changes is possible (committed to DB).

**`backend/app/services/dispute_service.py:181` — P2 — resolve_dispute emits no notifications to order parties**

`DisputeService.resolve_dispute` delegates to `CatalogService.resolve_dispute` which does notify both parties (lines 975–990 of catalog_service.py). However, the Dispute model `resolve_dispute` method wraps this call but adds no additional notifications of its own about the Dispute record's status change. This is technically correct (notifications come from catalog), but if the catalog resolve method is ever refactored, the notification chain is implicitly coupled.

**`backend/app/models/dispute.py:78` — P2 — Dispute.order and Dispute.admin relationships missing `lazy="raise"`**

```python
order: Mapped[ServiceOrder] = relationship("ServiceOrder")  # no lazy="raise"
admin: Mapped[User | None] = relationship("User", ...)     # no lazy="raise"
```

All other models in this codebase use `lazy="raise"` to prevent accidental N+1. The Dispute model omits this, so accessing `dispute.order` outside an eager-load context silently triggers a synchronous lazy load (which will raise `MissingGreenlet` in async SQLAlchemy context, crashing the endpoint). The `admin` relationship is properly loaded via `selectinload(Dispute.admin)` in `_load_dispute_by_id`, but `order` is never loaded — it's not in `DisputeOut` schema either, so no crash today, but it's a footgun if anyone adds `order` to the schema.

**`backend/app/services/catalog_service.py:992–1003` — P2 — list_disputed_orders missing freelancer/category selectinload**

`list_disputed_orders` does `selectinload(ServiceOrder.service)` but `ServiceOut` includes `freelancer: Optional[FreelancerBrief]`, `packages: list[ServicePackageOut]`, and `category: Optional[CategoryOut]` — all relationships on `Service` with `lazy="raise"`. Since these are `Optional` and default to `None` in the schema, pydantic's `from_attributes` mode will still attempt to access the attribute on the ORM object. With `lazy="raise"`, this would raise `sqlalchemy.exc.InvalidRequestError: 'Service.freelancer' is not available due to lazy='raise'`. This is a latent crash that fires only when `service.freelancer` (etc.) are set on the service object (i.e., always, since Service objects always have FK references). Should add `selectinload(ServiceOrder.service).selectinload(Service.freelancer)` etc.

**`backend/app/schemas/dispute.py:19` — P2 — evidence_files URLs not validated**

`evidence_files: list[str]` in `DisputeCreate` accepts arbitrary strings with no URL format validation. A malicious user could submit internal hostnames (`http://169.254.169.254/...`, `http://redis:6379/...`) as "evidence" links. These are stored in the DB and potentially displayed in admin panels or sent in notification content, enabling social-engineering attacks against admins. Should validate with `AnyHttpUrl` or an allowlist pattern.

**`backend/app/api/v1/endpoints/services.py:427` — P3 — reject_service logs rejection reason in access logs**

`reason: str = Query(...)` means the rejection reason is part of the request URL and will appear in nginx/uvicorn access logs in plain text. Rejection reasons may reference freelancer misconduct details or internal policy notes. Should be moved to a request body.

**`backend/app/services/catalog_service.py:800–801` — P2 — raise_dispute IDOR: no `SELECT FOR UPDATE` on order**

`order = await self._get_order(order_id)` is a plain `db.get()` with no row lock. Two concurrent requests to raise a dispute on the same order by the same client (or two clients if IDOR were possible) could both pass the status check before either commits. A `SELECT ... FOR UPDATE NOWAIT` would prevent this.

**`backend/app/services/dispute_service.py:165` — P3 — assign_admin allows support staff to self-assign without conflict check**

`assign_admin` sets `dispute.admin_id = admin.id` without checking if another admin is already assigned. If two admins simultaneously claim the same OPEN dispute, the last writer wins silently. Should check `if dispute.admin_id and dispute.admin_id != admin.id: raise ConflictError(...)`.
